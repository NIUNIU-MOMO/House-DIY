## 1. Git 安装（2026/05/20）

> 目的：通过 Homebrew 安装 Git，并配置 PATH 使 brew 版 Git 为默认

### 1.1 安装前状态

```bash
which git
# /usr/bin/git

git --version
# git version 2.50.1 (Apple Git-155)

which -a git
# /usr/bin/git

grep -n "homebrew/bin" ~/.zshrc
# 无 homebrew/bin 全局 PATH 配置
```

| 项 | 安装前 |
|----|--------|
| 默认 Git | Apple Git 2.50.1 |
| 路径 | `/usr/bin/git` |
| `~/.zshrc` PATH | 无 `/opt/homebrew/bin` 优先项 |

---

### 1.2 执行的命令

#### 1.2.1 安装 Git

```bash
brew install git
```

**安装结果：**

- Git 版本：`2.54.0`
- 安装路径：`/opt/homebrew/Cellar/git/2.54.0`
- 可执行文件：`/opt/homebrew/bin/git`
- 依赖：`libunistring 1.4.2`（自动安装）

#### 1.2.2 配置默认 PATH

在 `~/.zshrc` 文件**顶部**追加：

```bash
# Homebrew bin（默认优先使用 brew 安装的 git 等工具）
export PATH="/opt/homebrew/bin:$PATH"
```

使当前 shell 立即生效（新开终端会自动加载）：

```bash
source ~/.zshrc
```

#### 1.2.3 验证

```bash
which git
# /opt/homebrew/bin/git

git --version
# git version 2.54.0

which -a git
# /opt/homebrew/bin/git
# /opt/homebrew/bin/git
# /usr/bin/git
```

确认全局 Git 配置未变：

```bash
git config --global --list
```

输出（节选）：

```
user.name=niuniu
user.email=2810911384@qq.com
credential.https://github.com.helper=!/opt/homebrew/bin/gh auth git-credential
```

---

### 1.3 安装后状态

| 项 | 安装后 |
|----|--------|
| 默认 Git | Homebrew Git 2.54.0 |
| 默认路径 | `/opt/homebrew/bin/git` |
| Apple Git | 仍保留于 `/usr/bin/git`，未被删除 |
| 全局配置 | 未改动，与安装前一致 |

---

### 1.4 常用后续命令

```bash
# 查看当前使用的 Git
which git && git --version

# 临时使用 Apple Git
/usr/bin/git status

# 升级 brew Git
brew upgrade git

# 卸载 brew Git（会回退到 Apple Git）
brew uninstall git
```

---

### 1.5 说明

- `brew install git` **不会**删除或覆盖 `/usr/bin/git`（Apple Git），两者并存。
- PATH 中 `/opt/homebrew/bin` 排在前面时，终端默认使用 Homebrew Git。
- 已有仓库、提交历史、SSH 密钥、`~/.gitconfig` 均不受影响。

## 2. Python 虚拟环境 · 方案 A（2026/05/20）

> 目的：使用已有 Python 3.12 创建 `~/House-DIY-env` 虚拟环境  
> 参考：[04-本地环境检查与安装步骤 §3.1](./04-本地环境检查与安装步骤.md)

### 2.1 安装前状态

```bash
which python3 python3.12
# /usr/bin/python3
# /opt/homebrew/bin/python3.12

python3 --version
# Python 3.9.6

/opt/homebrew/bin/python3.12 --version
# Python 3.12.13

ls ~/House-DIY-env/bin/activate
# No such file or directory

grep -n "python@3.12" ~/.zshrc
# 无 python@3.12 PATH 配置
```

| 项 | 安装前 |
|----|--------|
| 系统 `python3` | 3.9.6（`/usr/bin/python3`） |
| Homebrew Python | 3.12.13（`/opt/homebrew/bin/python3.12`） |
| 虚拟环境 | 不存在 |
| `~/.zshrc` | 无 `python@3.12` PATH |

### 2.2 执行的命令

#### 2.2.1 配置 Python 3.12 PATH

在 `~/.zshrc` 中追加（位于 `/opt/homebrew/bin` 之后）：

```bash
# Python 3.12（House-DIY 后端虚拟环境）
export PATH="/opt/homebrew/opt/python@3.12/bin:$PATH"
```

使当前 shell 立即生效：

```bash
source ~/.zshrc
```

#### 2.2.2 创建虚拟环境并升级 pip

```bash
mkdir -p ~/House-DIY-env
/opt/homebrew/bin/python3.12 -m venv ~/House-DIY-env
source ~/House-DIY-env/bin/activate
pip install --upgrade pip
```

**执行结果：**

- 虚拟环境路径：`~/House-DIY-env`
- Python 版本：3.12.13
- pip：26.0 → 26.1.1

#### 2.2.3 验证

```bash
source ~/House-DIY-env/bin/activate
python --version
# Python 3.12.13

which python
# /Users/niuniu/House-DIY-env/bin/python

pip --version
# pip 26.1.1 from /Users/niuniu/House-DIY-env/lib/python3.12/site-packages/pip (python 3.12)
```

### 2.3 安装后状态

| 项 | 安装后 |
|----|--------|
| 虚拟环境 | `~/House-DIY-env` 已创建 |
| Python | 3.12.13（venv 内） |
| pip | 26.1.1 |
| 激活脚本 | `~/House-DIY-env/bin/activate` |

### 2.4 日常使用

**激活**虚拟环境（开发后端前）：

```bash
source ~/House-DIY-env/bin/activate
```

激活成功后，命令行前会出现 `(House-DIY-env)` 前缀，`python`、`pip` 指向虚拟环境内版本。

**退出**虚拟环境：

```bash
deactivate
```

退出后 `(House-DIY-env)` 前缀消失，`python`、`pip` 恢复为系统或 PATH 默认版本。直接关闭终端也会退出，下次需重新激活。

### 2.5 说明

- 方案 A 使用已有 Homebrew `python@3.12`，无需额外安装 Python 3.11。
- 系统 `/usr/bin/python3`（3.9.6）未被修改；激活 venv 后 `python` 指向虚拟环境。
- 后端依赖（`server/requirements.txt`）待项目代码就绪后再安装。

## 4. Redis · Docker 安装（2026/05/20）

> 目的：通过 Docker 运行 Redis 7.x，供任务队列使用（P2 可选）  
> 参考：[04-本地环境检查与安装步骤 §3.8](./04-本地环境检查与安装步骤.md)  
> 说明：文档原方案为 `brew install redis`；本次按需求改用 **Docker + Colima** 部署。

### 4.1 安装前状态

```bash
colima status
# colima is running

docker ps -a --filter name=house-diy-redis
# 无已有容器

brew list redis
# Error: No such keg（未通过 brew 安装）

which redis-cli
# redis-cli not found

lsof -i :6379
# 6379 端口空闲
```

| 项 | 安装前 |
|----|--------|
| Redis | 未安装 |
| Docker 运行时 | Colima 已运行 |
| 6379 端口 | 空闲 |

### 4.2 执行的命令

#### 4.2.1 确认 Docker 可用

若 `docker ps` 报 `docker.sock: no such file or directory`，需先启动 Colima：

```bash
colima start
colima status
docker ps
```

本次执行时 Colima **已在运行**，无需再次启动。

#### 4.2.2 拉取镜像并启动容器

```bash
docker pull redis:7

docker run -d \
  --name house-diy-redis \
  --restart unless-stopped \
  -p 6379:6379 \
  redis:7
```

**参数说明：**

| 参数 | 含义 |
|------|------|
| `--name house-diy-redis` | 容器名称，便于管理 |
| `--restart unless-stopped` | 宿主机重启后自动恢复（Colima 运行时可自启） |
| `-p 6379:6379` | 映射到本机 `127.0.0.1:6379` |
| `redis:7` | Redis 7.x 官方镜像 |

**执行结果：**

- 容器 ID：`2041dfb494ed`
- 镜像：`redis:7`（容器内 Redis **7.4.9**）

#### 4.2.3 验证

```bash
docker ps --filter name=house-diy-redis
```

```
CONTAINER ID   IMAGE     COMMAND                  CREATED   STATUS   PORTS                                         NAMES
2041dfb494ed   redis:7   "docker-entrypoint.s…"   ...       Up       0.0.0.0:6379->6379/tcp, [::]:6379->6379/tcp   house-diy-redis
```

```bash
docker exec house-diy-redis redis-cli ping
# PONG

docker exec house-diy-redis redis-server --version
# Redis server v=7.4.9 ...
```

### 4.3 安装后状态

| 项 | 安装后 |
|----|--------|
| 部署方式 | Docker（Colima） |
| 容器名 | `house-diy-redis` |
| Redis 版本 | 7.4.9 |
| 连接地址 | `redis://127.0.0.1:6379` |
| 健康检查 | `PONG` |

后端 `.env` 可配置（实现阶段）：

```bash
HOUSE_DIY_REDIS_URL=redis://127.0.0.1:6379/0
```

### 4.4 日常使用

```bash
# 查看状态
docker ps --filter name=house-diy-redis

# 健康检查
docker exec house-diy-redis redis-cli ping

# 进入 redis-cli
docker exec -it house-diy-redis redis-cli

# 停止 / 启动容器
docker stop house-diy-redis
docker start house-diy-redis

# 查看日志
docker logs house-diy-redis

# 删除容器（会丢失容器内未持久化数据）
docker stop house-diy-redis && docker rm house-diy-redis
```

**Colima 未运行时：**

```bash
colima start
docker start house-diy-redis   # 若容器已存在但未运行
```

### 4.5 说明

- MVP 阶段任务队列仍可用**内存队列**替代 Redis；安装 Redis 为 P2 可选，便于后续 Scheduler 对接。
- 当前未挂载数据卷，容器删除后数据会丢失；若需持久化，后续可加 `-v house-diy-redis-data:/data`。
- 本机未安装 `redis-cli` 客户端时，可通过 `docker exec house-diy-redis redis-cli` 操作。
- House-DIY 其他组件（oMLX、ComfyUI、FastAPI）不依赖 Docker；仅 Redis 通过 Docker 运行。

#### 6.7.5 内存管理

与 oMLX 不同，ComfyUI **无 Pin 机制**；模型在**首次跑 workflow** 时载入，之后可能缓存在内存中。

**释放已加载模型 / 内存（API）：**

```bash
curl -s -X POST http://127.0.0.1:8188/free \
  -H "Content-Type: application/json" \
  -d '{"unload_models": true, "free_memory": true}'
```

**中断当前生图：**

```bash
curl -s -X POST http://127.0.0.1:8188/interrupt
```

**与 oMLX 共存建议：** 批量出图前可卸载 oMLX 中非 Pin 模型；出图结束后调用 `/free` 释放 ComfyUI 内存，再跑 VLM / Embedding。

#### 6.7.7 HTTP API 端点速查

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/system_stats` | 系统内存 / 设备信息 |
| GET | `/models` | 模型类型列表 |
| GET | `/models/{folder}` | 指定目录下模型文件名（如 `diffusion_models`） |
| GET | `/object_info` | 全部节点 schema（构建 workflow 用） |
| GET | `/queue` | 当前队列状态 |
| GET | `/history` | 历史任务列表 |
| GET | `/history/{prompt_id}` | 指定任务结果与输出文件 |
| POST | `/prompt` | 提交 workflow 执行 |
| POST | `/queue` | 清空队列（`{"clear": true}`） |
| POST | `/interrupt` | 中断当前生图 |
| POST | `/free` | 卸载模型 / 释放内存 |
| POST | `/upload/image` | 上传输入图片（multipart） |
| GET | `/view?filename=...&type=output` | 下载输出图片 |

> ComfyUI 同时支持带 `/api` 前缀的路径（如 `/api/prompt`），与上表等价。

#### 6.7.8 提交 workflow（API 示例）

**步骤 1：** 在 GUI 中搭好 workflow，**Save (API Format)** 得到 JSON，例如 `workflows/test_api.json`。

**步骤 2：** 提交执行：

```bash
# client_id 可任意 UUID 字符串，用于 WebSocket 进度推送（HTTP 轮询可省略）
curl -s -X POST http://127.0.0.1:8188/prompt \
  -H "Content-Type: application/json" \
  -d @workflows/test_api.json
```

若 JSON 仅为 workflow 本体（`{"3": {...}, "4": {...}}`），需包装为：

```json
{
  "client_id": "house-diy-test",
  "prompt": { "...": "节点 ID → 节点定义" }
}
```

**步骤 3：** 响应中的 `prompt_id` 用于轮询结果：

```bash
PROMPT_ID="<上一步返回的 prompt_id>"

curl -s "http://127.0.0.1:8188/history/${PROMPT_ID}" | python3 -m json.tool
```

**步骤 4：** 从历史记录的 `outputs` 字段取得文件名，下载图片：

```bash
curl -s "http://127.0.0.1:8188/view?filename=ComfyUI_00001_.png&type=output&subfolder=" \
  -o /tmp/comfy_output.png
```

**上传输入图（ControlNet / img2img 场景）：**

```bash
curl -s -X POST http://127.0.0.1:8188/upload/image \
  -F "image=@/path/to/floorplan.png" \
  -F "type=input"
# 返回 JSON 含 name，填入 workflow 中 LoadImage 节点的 image 字段
```

| 首次出图很慢 | 正常，FLUX 全套首次加载约 14–18GB 内存 |
| 与 oMLX 同时卡死 | 串行调度；出图前 `/free` oMLX 非 Pin 模型，或 ComfyUI `/free` 释放 |
| FLUX ControlNet 报错 | 确认使用 XLabs v3 节点，非 SDXL ControlNet；缺节点见 §6.7.11 |
| 室内 LoRA 无效 | 提示词需包含触发词 **`interior`** |
| 自定义节点不生效 | 安装后需**完全重启** ComfyUI；检查 `custom_nodes/` 下是否有 `requirements.txt` |

#### 6.7.11 自定义节点安装

自定义节点目录：`~/ComfyUI/custom_nodes/`（ComfyUI 首次启动时会自动创建）。

**方式 A：ComfyUI-Manager（推荐）**

ComfyUI 官方内置 Manager，可在 GUI 中搜索、安装、更新节点。

```bash
cd ~/ComfyUI
source venv/bin/activate
pip install -r manager_requirements.txt
python main.py --listen 127.0.0.1 --port 8188 --enable-manager
```

浏览器打开 http://127.0.0.1:8188 ，在 Manager 面板中安装节点；安装完成后**重启 ComfyUI**。

常用启动参数（可选）：

| 参数 | 说明 |
|------|------|
| `--enable-manager` | 启用 ComfyUI-Manager |
| `--enable-manager-legacy-ui` | 使用旧版 Manager UI |
| `--disable-manager-ui` | 关闭 Manager UI，保留后台能力 |

**方式 B：手动 git clone**

```bash
cd ~/ComfyUI/custom_nodes
git clone https://github.com/<作者>/<节点仓库>.git

# 若节点自带 Python 依赖
cd ~/ComfyUI
source venv/bin/activate
pip install -r custom_nodes/<节点目录>/requirements.txt

# 重启 ComfyUI 后生效
```

**方式 C：更新已安装的自定义节点**

```bash
cd ~/ComfyUI/custom_nodes/<节点目录>
git pull
# 若 requirements.txt 有变更，重新 pip install -r requirements.txt
# 重启 ComfyUI
```

**House-DIY 相关说明：**

| 场景 | 建议 |
|------|------|
| FLUX ControlNet（depth / canny / hed） | 优先用 ComfyUI **原生 FLUX ControlNet 节点**；若 workflow 要求 XLabs 节点，通过 Manager 搜索 `XLabs` / `Flux ControlNet` 安装 |
| 室内 LoRA | 无需额外节点，原生 `LoraLoader` 即可 |
| 卸载节点 | 删除 `custom_nodes/<目录>` 并重启；Manager 中也可 Uninstall |

**验证节点已加载：**

```bash
# 启动日志中会出现 "Import times for custom nodes"
grep -i "custom nodes" ~/ComfyUI/comfyui.log

# 或通过 API 查看节点 schema 是否含新节点类名
curl -s http://127.0.0.1:8188/object_info | python3 -c \
  "import json,sys; d=json.load(sys.stdin); print(len(d), 'nodes')"
```

#### 6.7.12 macOS 开机自启（launchd，可选）

`nohup`（§6.7.1）适合临时后台运行；若需**登录后自动启动**、**崩溃后自动重启**，可用 macOS `launchd`。

**1. 创建 plist**

```bash
cat > ~/Library/LaunchAgents/com.house-diy.comfyui.plist <<'EOF'
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.house-diy.comfyui</string>
  <key>WorkingDirectory</key>
  <string>/Users/niuniu/ComfyUI</string>
  <key>ProgramArguments</key>
  <array>
    <string>/Users/niuniu/ComfyUI/venv/bin/python</string>
    <string>main.py</string>
    <string>--listen</string>
    <string>127.0.0.1</string>
    <string>--port</string>
    <string>8188</string>
  </array>
  <key>RunAtLoad</key>
  <true/>
  <key>KeepAlive</key>
  <true/>
  <key>StandardOutPath</key>
  <string>/Users/niuniu/ComfyUI/comfyui.log</string>
  <key>StandardErrorPath</key>
  <string>/Users/niuniu/ComfyUI/comfyui.log</string>
</dict>
</plist>
EOF
```

> 若使用 ComfyUI-Manager，在 `ProgramArguments` 的 `main.py` 后追加 `<string>--enable-manager</string>`。

**2. 加载 / 卸载服务**

```bash
# 加载（立即启动 + 登录自启）
launchctl load ~/Library/LaunchAgents/com.house-diy.comfyui.plist

# 查看状态
launchctl list | grep house-diy

# 停止并取消自启
launchctl unload ~/Library/LaunchAgents/com.house-diy.comfyui.plist
```

**3. 验证**

```bash
curl -s -o /dev/null -w "HTTP %{http_code}\n" http://127.0.0.1:8188/
tail -f ~/ComfyUI/comfyui.log
```

**注意：**

- launchd 与手动 `python main.py` / `nohup` **不要同时运行**，否则会争用 8188 端口。
- ComfyUI 生图占用大量内存；开机自启适合开发机常开场景，与 oMLX 并存时注意 §6.7.5 内存调度。
- 修改 plist 后需先 `unload` 再 `load` 才生效。

#### 6.7.13 队列与历史管理

**清空排队中的任务（不影响已完成的 history）：**

```bash

  -H "Content-Type: application/json" \
```

```bash
  -H "Content-Type: application/json" \
```

```bash

  -H "Content-Type: application/json" \
  -d '{"delete": ["<prompt_id>"]}'
```

**查看当前队列详情：**

```bash

```
